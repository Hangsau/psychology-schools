## 步驟 4｜預期風險

### 風險一：背景引擎可能與人工寫入產生競爭

- **發生步驟**：步驟 1 或步驟 3。
- **具體原因**：本專案的 m3 引擎仍在運轉，`naikan-therapy` 在目前 `STATUS.md` 中仍是 queued；若引擎在人工寫入前後處理同一 slug，可能覆蓋人工內容，或產生另一份未校核草稿。
- **類型**：狀態假設、共用檔案副作用。

### 風險二：內觀療法史料與實證資料可能不足，導致過度確定或捏造

- **發生步驟**：步驟 2、步驟 3。
- **具體原因**：內觀療法的英文學術資料相對有限，部分資料混合宗教介紹、臨床機構宣傳與二手概述；吉本伊信生平、起源年代、三項反省的翻譯、臨床成效與後續發展可能存在來源差異。
- **類型**：外部依賴、研究資料不完整、事實錯誤。

### 風險三：書目版權狀態與出版資訊不一定能可靠判定

- **發生步驟**：步驟 2、步驟 3。
- **具體原因**：日文原典、英文譯本、中文譯本與機構教材的出版年代、版本、編者與著作權狀態可能不同；若把作者原著的公有領域狀態直接套到編輯版、翻譯版或重印版，會造成錯誤的 PD/© 標記。
- **類型**：外部依賴、版權與出版資料歧異。

### 4b. 結構性風險檢查

本次不涉及程式碼、API、資料庫、設定或服務，因此以下項目均不適用：

- **並發**：不新增程式 lock 或跨進程邏輯；但人工檔案與背景引擎競爭已在風險一處理。
- **邊界輸入**：不新增輸入處理程式。
- **持久化/schema migration**：不修改 schema 或資料庫。
- **外部輸入驗證**：研究資料會人工交叉核對，不將外部文字直接當作程式輸入。
- **命令注入**：不組合 SSH、SQL、shell 或動態命令。
- **跨狀態一致性**：不修改 `status`、`review_state` 或其他狀態欄位。

## 步驟 5｜風險預案

### 風險一的預案：引擎競爭或覆蓋

- **提前偵測**
  - 寫入前讀取 `STATUS.md`，確認 `naikan-therapy` 的狀態。
  - 搜尋 engine log 是否已出現 `naikan-therapy` 的處理紀錄。
  - 寫入後立即讀回檔案，確認大小、開頭與八段標題沒有被改寫。
- **發生後處理**
  - 若寫入前發現引擎正在處理，先不覆蓋現有檔案，停止在確認點回報。
  - 若寫入後發現檔案被覆蓋，重新讀取實際內容並以本次已核對的綜述重新寫入，之後再次執行完整性檢查。
  - 不刪除或重設引擎狀態，不使用破壞性 Git 指令處理衝突。

### 風險二的預案：資料不足或學術說法分歧

- **提前偵測**
  - 每個關鍵歷史、人物、臨床成效宣稱至少尋找兩個獨立可靠來源。
  - 特別標記只有機構網站、百科或科普文章支持的內容。
  - 將無法交叉確認的內容列入研究缺口清單，而不是補猜數字或年代。
- **發生後處理**
  - 降低確定性標記，例如由 `🟢` 改為 `🟡` 或 `🟠`。
  - 在 §8 並列不同說法，說明爭議點與需要查閱的原典或資料庫。
  - 刪除無法找到真實出處的具體宣稱。

### 風險三的預案：書目或版權標示不確定

- **提前偵測**
  - 逐項確認書名、作者、出版年與版本，不把搜尋結果中的簡略書名直接當正式書目。
  - 分開判斷原著、翻譯、編輯本與重印版。
  - 無法確認時不填精確出版者、卷次或法律結論。
- **發生後處理**
  - 將不確定狀態標為 `🟠` 或 `🔴`，並寫「資料待補」。
  - 只列出已確認的書目，不為了湊足數量補入可疑著作。
  - 以「PD/© 需依司法管轄區與版本確認」的保守表述取代武斷判定。
